# DevSecOps란?
DevSecOps는 Development(개발)·Security(보안)·Operations(운영)의 합성어로, 소프트웨어 개발 수명주기(SDLC) 전 단계에 보안을 통합하는 개발·운영 문화이자 방법론이다. DevOps가 속도와 자동화를 중심으로 개발과 운영 사이의 벽을 허무는 것이라면, DevSecOps는 여기에 “보안을 처음부터 끝까지 같이 넣자”는 확장 개념이다.

## 핵심 개념

- 보안의 “Shift Left”  
  - 보안을 개발 마지막 단계나 배포 후에 붙이는 것이 아니라, 요구사항·설계·코딩·빌드·테스트·배포·운영 모든 단계에 내재화한다.
  - 초기 설계 때부터 위협 모델링, 안전한 아키텍처 검토, 보안 요구사항 정의를 포함한다.

- 보안의 공동 책임(Shared Responsibility in SDLC)  
  - 보안 전담팀만이 아니라 개발자·운영자·보안 담당자 모두가 보안에 대한 책임을 공유한다.
  - 코드 품질과 동일하게 “보안 품질”을 팀 공통 KPI로 다루며, 피처 개발과 보안 이슈 해결을 동시에 고려한다.

- 자동화된 보안 통합  
  - CI/CD 파이프라인에 정적/동적 분석(SAST/DAST), 의존성 취약점 스캔, 컨테이너 이미지 스캔, IaC 스캔 등을 자동으로 포함한다.
  - 반복 가능한 자동화 스캔·테스트로, 코드 커밋·빌드·배포 시마다 보안 검사가 같이 실행되도록 한다.

- 지속적인 보안(Continuous Security)  
  - 배포 이후에도 모니터링, 로깅, 런타임 보안, 취약점 관리, 패치 자동화 등을 통해 보안을 계속 유지·강화한다.
  - 이상 탐지, 로그 분석, 정책 위반 감시를 통해 운영 중 발생하는 위협에 상시 대응한다.

## DevOps와 DevSecOps 비교

| 구분 | DevOps | DevSecOps |
|------|--------|-----------|
| 목표 | 빠른 배포, 자동화, 협업 | 빠른 배포 + “처음부터” 안전한 소프트웨어 제공 |
| 보안 위치 | 주로 테스트/배포 이후 별도 단계 | SDLC 전 단계에 내장(요구사항~운영) |
| 책임 | 주로 Dev + Ops | Dev + Sec + Ops 모두 공동 책임 |
| 파이프라인 | 빌드·테스트·배포 자동화 중심 | 빌드·테스트·배포 + 보안 스캔/검증 자동화 |



## 주요 실천 요소

- 프로세스/문화  
  - 개발·보안·운영 팀 간 사일로 제거, 공통 목표·공동 의사결정 구조 수립.
  - 개발자 보안 교육, 보안 코딩 가이드, 보안 리뷰 문화 정착.

- 기술/도구  
  - 코드/라이브러리 취약점 스캐너(SCA, SAST), 동적 분석(DAST), IaC 스캐너, 컨테이너·쿠버네티스 보안 도구, 비밀관리(Secrets Management) 등 활용.
  - CI/CD 시스템(Jenkins, GitLab CI, GitHub Actions, AWS CodePipeline 등)과 보안 도구의 연계로 파이프라인에 보안 단계를 삽입.

- 거버넌스·컴플라이언스  
  - 코드 수준에서 정책(Policy as Code)으로 보안·규정 요구사항을 자동 검증.
  - 표준 프레임워크(NIST, CIS, ISO 등)에 따라 로그·감사·리포팅 체계를 자동화.

## 기대 효과

- 배포 속도 유지하면서도 보안 사고·취약점 감소.
- 보안 이슈를 개발 초기에 발견·해결해 비용과 리스크를 크게 줄임.
- 규정 준수 요구사항 대응이 쉬워지고, 감사·보안 점검 준비 부담이 감소.

정리하면, DevSecOps는 “DevOps의 민첩성과 자동화에 ‘보안 내재화’를 더한 현대적 소프트웨어 개발·운영 방식”으로, 클라우드·마이크로서비스·빠른 릴리즈가 기본인 환경에서 사실상 필수적인 보안 접근법이라고 볼 수 있다.
