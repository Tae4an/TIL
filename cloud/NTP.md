# NTP(Network Time Protocol)란
## 1. 개념 요약
NTP(Network Time Protocol)는 **네트워크로 연결된 장비들의 시계를 협정세계시(UTC) 기준으로 정확하게 동기화하는 프로토콜**이다. 1985년부터 운용 중인 가장 오래된 인터넷 프로토콜 중 하나로, **UDP 123번 포트**를 사용하며 수 밀리초(ms) 이하의 정확도로 시간을 맞춘다.

***
## 2. 동작 원리
NTP는 **클라이언트-서버 모델**로 동작한다. 클라이언트가 NTP 서버에 시간 요청 패킷을 보내면, 서버는 현재 시각을 응답한다. 이때 클라이언트는 **패킷 왕복 지연(Round-trip Delay)과 시간 오차(Offset)를 함께 계산**해, 네트워크 지연을 보정한 뒤 자신의 시계를 정밀하게 조정한다. 시계를 급격히 변경하지 않고 **슬루잉(Slewing)** 방식으로 천천히 교정해 시스템 안정성을 유지한다.
***
## 3. Stratum 계층 구조
NTP는 시간 출처의 신뢰도에 따라 계층(Stratum)을 나눈다.

- **Stratum 0**: 원자시계·GPS 수신기 등 물리적 기준 시계 (네트워크에 직접 연결 안 됨)
- **Stratum 1**: Stratum 0에 직접 연결된 최상위 NTP 서버 (가장 정확, Primary Server)
- **Stratum 2**: Stratum 1 서버에서 동기화를 받는 서버
- **Stratum 3~15**: 상위 계층에서 순차적으로 동기화를 받는 서버·클라이언트
- **Stratum 16**: 동기화 미완료 상태 (unreachable)

계층이 낮을수록 원자시계에 가깝고 정확도가 높으며, 대부분의 일반 서버는 Stratum 3~4 수준에서 운용된다.

***
## 4. 왜 시간 동기화가 중요한가
분산 시스템에서 서버 간 시간이 어긋나면 심각한 문제가 발생한다.

- **로그 분석·트러블슈팅**: 서버 간 로그 시각이 달라 장애 원인 추적이 불가능해짐
- **인증·보안**: TLS 인증서 유효성 검사, Kerberos 티켓, JWT 만료 등 시간 기반 인증이 실패
- **분산 DB·트랜잭션**: Kafka, Cassandra, DynamoDB 등은 시간 기반 이벤트 순서에 의존
- **컴플라이언스**: 금융·의료 업계는 정확한 타임스탬프 기록이 규제 요구사항

***
## 5. NTP vs PTP 비교
| 항목 | NTP | PTP (IEEE 1588) |
|---|---|---|
| 정확도 | 수 ms~수십 ms | 나노초(ns) 수준 |
| 사용 환경 | 일반 인터넷·데이터센터 | 통신·금융·산업제어 |
| 프로토콜 | UDP 123 | UDP/Ethernet |
| 하드웨어 지원 | 불필요 | 하드웨어 타임스탬프 필요 |
| 도입 난이도 | 매우 쉬움 | 상대적으로 복잡 |

***
## 6. AWS에서의 NTP
AWS는 EC2 인스턴스에 기본적으로 **Amazon Time Sync Service**를 제공한다. 169.254.169.123 IP로 접근 가능한 내부 NTP 서버로, AWS 인프라 내 원자시계 기반 Stratum 1 서버에 연결된다. 인터넷 외부 NTP 서버 없이도 정확한 시간 동기화가 가능하며, EC2 내 chrony 또는 ntpd 설정에서 기본 서버로 권장된다.

***
## 7. 한 줄 정리
NTP는 **원자시계를 최상위로 하는 Stratum 계층 구조를 통해, 네트워크의 모든 서버·장비 시계를 UTC 기준 수 밀리초 이하로 동기화하는 인터넷 표준 시간 프로토콜**이며, 로그·인증·분산 트랜잭션 등 IT 인프라 전반의 신뢰성 기반이 된다.
