# Proxy란?
## 개념 정의
프록시(Proxy)는 **클라이언트와 서버 사이에 위치해서, 한쪽의 요청을 대신 받아 다른 쪽에 전달하는 중개 서버·소프트웨어**를 의미한다. "대리인"이라는 어원 그대로, 클라이언트·서버 중 한쪽이 상대방과 직접 통신하는 대신 프록시를 통해 간접적으로 통신하는 구조임.

***
## 프록시가 하는 일 한 줄로
"클라이언트와 서버 사이에서 트래픽을 가로채고, 필요에 따라 **캐싱·필터링·보안 검사·접근 통제·부하 분산 등**을 수행하고 요청을 중계하는 중간자 역할"이라고 보면 된다.

***
## 주요 유형
### 1. Forward Proxy (포워드 프록시)
- 클라이언트 앞에 위치해 클라이언트를 "대신해서" 인터넷/서버에 요청을 보낸다.
- 서버 입장에서는 실제 클라이언트 IP가 아닌 프록시 IP만 보인다.
- 주요 목적
  - 클라이언트 IP 익명화 및 프라이버시 보호.
  - 특정 사이트 접근 제어 (기업 내 웹 필터링, SWG 등에 활용).
  - 자주 요청되는 콘텐츠 캐싱으로 성능 향상.
  - 망 분리 환경에서 내부 클라이언트가 인터넷에 접근하는 게이트웨이.
### 2. Reverse Proxy (리버스 프록시)
- 서버 앞에 위치해 클라이언트의 요청을 대신 받아 내부 서버로 전달한다.
- 클라이언트 입장에서는 실제 내부 서버 주소를 알 수 없고, 프록시 서버와만 통신한다.
- 주요 목적
  - 내부 서버 IP·구조 은닉 (보안 강화).
  - 로드 밸런싱: 여러 내부 서버로 요청 분산.
  - SSL 터미네이션(TLS 처리 오프로드).
  - 캐싱·압축·콘텐츠 최적화.
  - nginx, HAProxy, AWS ALB 같은 서비스가 대표적인 Reverse Proxy임.
### 3. Transparent Proxy (투명 프록시)
- 클라이언트가 프록시 존재를 인식하지 못하도록, 사용자 설정 없이 네트워크 레벨에서 트래픽을 자동으로 가로채는 프록시.
- 보통 방화벽·SWG에서 클라이언트 설정 없이도 웹 트래픽을 검사·필터링하기 위해 사용.
### 4. SSL/HTTPS 프록시 (인터셉팅 프록시)
- TLS/SSL로 암호화된 HTTPS 트래픽을 복호화해 내용을 검사한 뒤, 다시 암호화해서 목적지로 전달하는 방식.
- SWG, DLP, CASB, 기업 보안 솔루션에서 암호화된 트래픽 내 위협·데이터 유출을 탐지하기 위해 사용.
- 중간에서 인증서를 재서명(Man-in-the-Middle 방식)하므로, 클라이언트에 신뢰 CA를 배포해야 한다.

***
## 포워드 vs 리버스 프록시 한눈에 비교
| 항목 | Forward Proxy | Reverse Proxy |
|------|--------------|---------------|
| 위치 | 클라이언트 앞 | 서버 앞 |
| 대리 대상 | 클라이언트를 대신 | 서버를 대신 |
| 숨겨지는 것 | 클라이언트 IP (서버가 모름) | 내부 서버 IP (클라이언트가 모름) |
| 주 목적 | 웹 필터링, 접근 제어, 캐싱, 클라이언트 익명화 | 로드 밸런싱, 내부 서버 보호, SSL 처리, 캐싱 |
| 대표 사용 예 | 기업 웹 프록시, SWG, VPN 게이트웨이 | nginx, ALB, CloudFront, API Gateway |

***
## 프록시와 보안 솔루션의 관계
SWG, CASB, Zscaler ZIA, Web Filtering 같은 보안 솔루션들은 대부분 **풀 프록시(Full Proxy) 아키텍처** 기반으로 동작한다. 즉, 클라이언트와 서버 사이에서 프록시로 트래픽을 전부 가로채서 검사·정책 적용을 하기 때문에,  
"프록시 = SWG/보안 게이트웨이의 핵심 동작 원리"라고 이해하면 된다.
