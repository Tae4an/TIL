# PDP & PEP란?
## 개념 정의
- **PDP (Policy Decision Point)**  
  접근 요청이 들어왔을 때, **정책과 각종 정보를 바탕으로 “허용/차단/조건부 허용” 같은 결정을 내리는 논리적 지점(엔진/서비스)** 이다.  
- **PEP (Policy Enforcement Point)**  
  PDP가 내린 결정을 실제 트래픽·세션·API 호출에 적용해 **접근을 허용하거나 차단하는 “문지기(게이트웨이)” 역할의 지점** 이다.

쉽게 말해, **PDP는 심판, PEP는 문지기**라고 보면 된다.  

***
## PDP (Policy Decision Point)
- 역할  
  - “이 요청을 허용할까, 말까? 어떤 조건으로 허용할까?”를 판단하는 **정책 결정 두뇌**.  
  - 정책 엔진(Policy Engine, PE), 정책 관리자(Policy Administrator, PA) 등을 포함한 제어 영역으로 구현되는 경우가 많다.  
- 입력으로 활용하는 정보 예시  
  - 사용자·디바이스 정보, 인증·MFA 결과  
  - IP/위치, 시간대, 리스크 점수, 행위 분석 결과  
  - 자원 분류(중요도), 규정·컴플라이언스 요구사항 등  
- 출력  
  - Allow / Deny / Allow with conditions(예: MFA 추가, 읽기 전용 등)  
  - 세션 시간, 접근 범위, 의무 사항(로그 남기기 등)  

제로 트러스트 아키텍처에서 PDP는 "항상 검증, 상황별 동적 결정"의 핵심 두뇌 역할을 맡는다.  

***
## PEP (Policy Enforcement Point)
- 역할  
  - 실제로 **요청을 가로채서 PDP에 물어보고, 나온 결과를 따라 허용·차단·제한을 수행하는 지점**.  
  - “문 앞에서 사람을 막거나 들이는 문지기”에 해당한다.  
- 위치 예시  
  - API Gateway, 프록시(SWG), ZTNA 게이트웨이, 리버스 프록시  
  - 애플리케이션 게이트웨이, 서비스 메시 사이드카, 에이전트 등  
- 하는 일  
  - 클라이언트의 요청을 받아 PDP에 질의(Authorize?)  
  - PDP 응답에 따라 세션 생성/차단, 리다이렉트, 토큰 검증, 로깅 등 수행  
  - 세션 중에도 재평가 요청을 보내거나, PDP 신호에 따라 세션을 강제 종료하기도 한다.  

***
## PDP vs PEP 한눈에
| 항목 | PDP | PEP |
|------|-----|-----|
| 풀네임 | Policy Decision Point | Policy Enforcement Point |
| 역할 | 정책 평가·결정 (두뇌, 심판) | 정책 집행 (게이트, 문지기) |
| 위치 | 제어 영역(Control Plane) 서비스/엔진 | 데이터 경로(Data Plane) 인라인/에이전트 |
| 주요 기능 | Context 수집, 정책 평가, 허용/차단/조건부 결정 | 요청 가로채기, PDP 질의, 결과에 따른 허용/차단/리다이렉트 |
| 예시 | 중앙 권한 서버, OPA/ABAC 엔진, IAM 정책 엔진 | API Gateway, 프록시, ZTNA 게이트웨이, 서비스 메시 프록시 |

***
## XACML·제로 트러스트 문맥에서
- XACML, 제로 트러스트 가이드, NIST 800‑207 등에서는  
  - **PDP**: 정책을 평가하는 엔진/서비스  
  - **PEP**: 접근 요청을 가로채 PDP에 보낸 뒤, 결과를 적용하는 인라인 컴포넌트로 정의한다.  

제로 트러스트에서는 **PDP와 PEP를 분리**해서,  
- PDP: 지속적으로 컨텍스트·위협 정보를 반영해 동적으로 판단  
- PEP: 네트워크·애플리케이션 가장자리에서 그 결정을 계속 강제하는 구조를 권장한다.
