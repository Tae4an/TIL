# NAT at Firewall
## 개념 정의
“NAT at firewall”은 **방화벽 장비가 트래픽 필터링(허용/차단)뿐 아니라, 동시에 NAT(Network Address Translation)까지 수행하는 구조**를 말한다.  
요즘 대부분의 엔터프라이즈/UTM/차세대 방화벽들은 “방화벽 + NAT 게이트웨이” 역할을 같이 하므로, 방화벽 인터페이스에서 출발지/목적지 IP·포트 변환을 함께 처리한다.

***
## 방화벽에서 NAT가 하는 일
- 사설 → 공인 IP 변환 (Source NAT, SNAT)  
  - 내부 사설망(예: 10.x, 172.16/12, 192.168/16)의 출발지 주소를 방화벽의 공인 IP로 바꿔 인터넷으로 나가게 함.  
  - PAT(Port Address Translation)로 여러 내부 IP를 하나의 공인 IP + 서로 다른 포트로 공유.

- 외부 → 내부 서버 노출 (Destination NAT, DNAT)  
  - 외부에서 방화벽 공인 IP:포트로 들어오는 트래픽을 내부 서버 IP:포트로 변환.  
  - 예: `203.0.113.10:443 → 10.0.0.10:443`으로 웹 서버 퍼블리싱.

- Static NAT / Dynamic NAT / PAT  
  - Static: 1:1 고정 매핑 (특정 서버 항상 같은 공인 IP).  
  - Dynamic: 공인 IP 풀에서 동적으로 할당.  
  - PAT(NAPT): 여러 사설 IP를 하나의 공인 IP+서로 다른 포트로 동시에 사용.

***
## “방화벽 + NAT” 동작 흐름
일반적인 순서는 대략 다음과 같이 이해하면 된다.

1. 패킷이 방화벽에 도착  
2. NAT 규칙으로 IP/포트 변환 (SNAT 또는 DNAT)  
3. 변환된 헤더 기준으로 방화벽 정책(ACL/보안정책) 평가  
4. 허용되면 세션 생성, 상태 테이블에 NAT 매핑과 세션 상태 기록  
5. 응답 패킷이 들어올 때 상태 테이블을 보고 역방향 NAT 및 허용 처리

중요한 점은 **NAT 자체는 “주소 변환”만 할 뿐, 보안 정책은 방화벽 룰과 상태 기반 검사에서 담당**한다는 것.  
그래서 “NAT = 방화벽”은 아니지만, 실제 장비에서는 둘이 한 박스 안에서 함께 움직인다.

***
## 왜 방화벽에서 NAT를 같이 하느냐
- 외부에 노출되는 주소를 최소화하고, 내부 구조를 숨기기 위해  
- 공인 IP를 절약하기 위해 (여러 내부 장비를 하나/소수의 공인 IP로 묶기)  
- 인바운드/아웃바운드 트래픽을 **단일 지점(방화벽)** 에 집중시켜,  
  - 주소 변환 + 필터링 + 로깅 + IPS/AV 검사까지 한 번에 처리하기 위해

실무에서 “Firewall NAT Rule 설정”이라고 하면,  
이 방화벽 장비 위에서 SNAT/DNAT/PAT 규칙을 정의해 **어떤 트래픽을 어떤 주소로 바꾸어 내보낼지/받을지**를 세밀하게 설계하는 작업을 의미한다.
