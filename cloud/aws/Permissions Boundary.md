# AWS 권한 경계(Permissions Boundary)란?

## 개념 정의

AWS IAM 권한 경계(Permissions Boundary)는 **IAM 사용자·역할에 부여할 수 있는 최대 권한의 상한선**을 정의하는 기능이다. IAM 정책(Permissions Policy)에서 허용하는 권한과 권한 경계에서 허용하는 권한의 **교집합(둘 다 허용하는 범위)**만 실제로 실행할 수 있다. 권한 경계는 권한을 부여하는 것이 아니라 **권한을 제한하는 안전장치** 역할을 한다.

쉽게 말해, "관리자가 개발자에게 '이 정도 권한까지는 스스로 설정해도 되지만, 절대 이 선을 넘으면 안 된다'는 안전 울타리를 설정해주는 기능"이다.

## 작동 원리

### 권한 평가 방식

```
실제 실행 가능한 권한 = IAM 정책(Permissions Policy) ∩ 권한 경계(Permissions Boundary)
```

**예시**  
```
IAM 정책: S3, EC2, Lambda 모든 권한 허용 (AdministratorAccess)
권한 경계: S3, DynamoDB만 허용
실제 권한: S3만 실행 가능 (EC2, Lambda 차단)
```

### 설정 위치

**IAM 사용자/역할 생성·수정 시**  
```
Permissions (탭) → Permissions boundary → Set permissions boundary
```

## 주요 활용 사례

### 1. 개발자 권한 위임 (권장 시나리오)
```
상황: 개발자가 Lambda 개발을 위해 필요한 리소스 권한을 스스로 설정하고 싶지만, 프로덕션 S3나 다른 개발자의 리소스에 접근하지 못하게 하고 싶음

해결:
- IAM 정책: AdministratorAccess (개발자가 자유롭게 권한 설정 가능)
- 권한 경계: 개발자 권한 범위 제한 정책 부여 (자신의 리소스만 접근 가능)
```

### 2. 신규 직원/인턴 보호
```
상황: 신입 개발자가 실수로 과도한 권한을 자신에게 부여하거나, 다른 리소스를 삭제하지 못하게 하고 싶음

해결:
- 신입 계정에 제한된 권한 경계 설정
- 신입이 설정한 모든 역할·정책도 권한 경계에 의해 제한됨
```

### 3. CI/CD 파이프라인 권한 제한
```
상황: CodePipeline이 배포 시 필요한 권한만 사용하고, 개발자가 실수로 과도한 권한을 설정하지 못하게 하고 싶음

해결:
- CodePipeline 역할에 배포에 필요한 최소 권한만 허용하는 권한 경계 설정
```

### 4. 위임된 관리자 권한 제한
```
상황: 부서별 IAM 관리자를 두되, 조직 전체 권한을 가지지 못하게 하고 싶음

해결:
- 부서별 권한 경계 정책 설정
- IAM 관리자가 생성한 사용자·역할도 동일한 경계 적용
```

## 설정 예시

### 권한 경계 정책 생성
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:*",
        "dynamodb:*",
        "lambda:*"
      ],
      "Resource": "*"
    }
  ]
}
```

### 사용자에 권한 경계 설정
```
IAM 사용자 선택 → Permissions 탭 → Add permissions boundary
→ 위에서 생성한 정책 선택 → Set boundary
```

### IAM 정책과 권한 경계 동시 설정
```
IAM 정책: s3:*, ec2:*, iam:*
권한 경계: s3:* 만 허용
결과: s3:* 만 실행 가능 (ec2, iam 차단)
```

## IAM 정책 vs 권한 경계 비교

| 항목 | IAM 정책 (Permissions Policy) | 권한 경계 (Permissions Boundary) |
|------|-------------------------------|---------------------------------|
| 목적 | 권한 부여 | 권한 제한 (최대 범위 설정) |
| 동작 | 허용된 액션 실행 | 허용 범위 초과 시 차단 |
| 설정 위치 | 사용자·역할에 직접 부착 | 사용자·역할의 Permissions boundary 설정 |
| 권한 평가 | 실제 실행 권한 결정 | 최대 허용 범위 제한 |
| 누가 설정 | 관리자, 개발자 | 중앙 IAM 관리자만 |

## 설정 시 주의사항

### 1. 권한 경계는 **권한을 부여하지 않는다**
```
잘못된 예:
권한 경계만 설정 → 권한 없음 (실행 불가)

올바른 예:
IAM 정책 + 권한 경계 설정 → 교집합 권한 실행
```

### 2. 권한 경계는 **계승된다**
```
관리자가 개발자 A에게 권한 경계 설정
개발자 A가 역할 B 생성 → 역할 B에도 동일한 권한 경계 자동 적용
```

### 3. 권한 경계 **제거 불가 권한 설정**
```
관리자는 개발자가 권한 경계를 제거하지 못하게 iam:DeleteRolePermissionsBoundary, iam:PutRolePermissionsBoundary 등에 조건 추가
```

## 실제 적용 예시

### 개발자 권한 위임 시나리오
```
1. 중앙 IAM 팀이 "DeveloperBoundary" 정책 생성
   - 개발자 리소스(S3, Lambda, EC2)만 허용
   
2. 개발자 계정 생성
   - IAM 정책: PowerUserAccess
   - 권한 경계: DeveloperBoundary
   
3. 개발자가 역할 생성
   - 개발자 역할도 DeveloperBoundary 상속
   
4. 결과
   - 개발자는 필요한 권한 설정 가능
   - 프로덕션 리소스 접근 불가
```

## 정리

AWS 권한 경계(Permissions Boundary)는 IAM 사용자·역할의 **최대 권한 상한선을 설정**하는 안전장치로, IAM 정책과 권한 경계의 **교집합만 실제 실행**되며, 개발자 권한 위임, 신입 직원 보호, CI/CD 파이프라인 제한, 위임된 관리자 통제에 활용되고, 권한을 부여하는 것이 아니라 **권한을 제한**하는 역할을 하며, 계승되어 개발자가 생성한 역할에도 동일한 경계가 적용되며, 중앙 IAM 팀이 안전하게 권한 위임을 할 수 있게 해준다.
