# AWS SLA란?

## 개념 정의

AWS SLA(Service Level Agreement, 서비스 수준 계약)는 AWS가 고객에게 제공하는 **각 서비스의 가동 시간(Uptime) 보장 수준**을 명시한 공식 계약이다. SLA에는 월간 가동률 목표, 가동률 미달 시 보상(서비스 크레딧), 크레딧 청구 절차 등이 정의되어 있으며, AWS는 300개 이상의 유료 서비스마다 별도의 SLA를 제공한다.

쉽게 말해, "AWS가 '우리 서비스는 한 달에 99.9% 이상 가동하겠다'고 약속하고, 그 약속을 지키지 못하면 요금 일부를 돌려주는 보증서"다.

## 핵심 개념

### 월간 가동률 (Monthly Uptime Percentage)
- 특정 서비스가 한 달 동안 정상 작동한 시간의 비율
- 예: 99.99% = 한 달에 최대 4.32분 다운타임 허용, 99.9% = 한 달에 최대 7시간 12분 다운타임 허용

### 서비스 크레딧 (Service Credit)
- SLA 기준 미달 시 AWS가 제공하는 보상
- 월 사용 요금의 일정 비율을 다음 청구서에서 차감 (환불 아님)
- 크레딧 비율은 가동률 저하 정도에 따라 차등 적용

### SLO (Service Level Objective, 서비스 수준 목표)
- SLA 내에서 정의된 구체적인 지표 (응답 시간, 가동률 등)
- 예: "EC2 Multi-AZ 배포 시 99.99% 가동률"

## 주요 AWS 서비스별 SLA

### Amazon EC2

**Region-Level SLA (리전 수준)**
- Multi-AZ 배포 시: 99.99% 가동률 보장
- 미달 시 서비스 크레딧:
  - 99.99% 미만 ~ 99.0% 이상: 10%
  - 99.0% 미만 ~ 95.0% 이상: 30%
  - 95.0% 미만: 100%

**Instance-Level SLA (인스턴스 수준)**
- 단일 인스턴스: 99.5% 가동률 보장
- 미달 시 서비스 크레딧:
  - 99.5% 미만 ~ 99.0% 이상: 10%
  - 99.0% 미만 ~ 95.0% 이상: 30%

### Amazon S3
- 99.9% 가동률 보장
- 미달 시 서비스 크레딧:
  - 99.9% 미만 ~ 99.0% 이상: 10%
  - 99.0% 미만: 25%
- 내구성(Durability)은 별도로 99.999999999% (11 nines) 보장

### Amazon EBS
- 단일 AZ 내 99.99% 가용성 보장
- 볼륨이 EC2 인스턴스에 연결되어 있을 때 적용

### Amazon RDS
- Multi-AZ 배포 시: 99.95% 가동률 보장
- 단일 AZ 배포 시: 별도 SLA 없음 (Multi-AZ 권장)

## 서비스 크레딧 청구 방법

### 청구 조건
1. AWS Support Center에서 케이스 오픈
2. 제목에 "SLA Credit Request" 명시
3. 장애 발생 후 **2개 청구 사이클 이내** 청구 필요
4. 영향받은 리전, 서비스, 기간, 가동률 증거 제출

### 크레딧 적용 방식
- 현금 환불 아님
- 다음 청구서에 크레딧으로 적용
- 영향받은 서비스·리전의 월 사용 요금 기준 계산
- Reserved Instance 선불금 등은 크레딧 계산에서 제외

### 주의사항
- AWS가 자동으로 크레딧을 주지 않음, **고객이 직접 청구해야 함**
- Region-Level SLA와 Instance-Level SLA 중 하나만 청구 가능 (중복 불가)
- 고객의 잘못(예: 잘못된 설정, 네트워크 문제)으로 인한 다운타임은 제외

## SLA의 한계

### SLA가 보장하지 않는 것
- **성능**: IOPS, 지연 시간, 처리량은 SLA 범위 밖
- **데이터 손실**: 가용성만 보장, 데이터 백업은 고객 책임
- **사용자 오류**: 잘못된 구성, 보안 설정 미흡 등은 고객 책임
- **예약 인스턴스**: RI 선불금은 크레딧 계산에서 제외

### 실제 고가용성 달성 방법
SLA만으로는 완전한 고가용성을 보장할 수 없으므로, 아래 아키텍처 패턴을 적용해야 한다:

- **Multi-AZ 배포**: 단일 AZ 장애 대비
- **Multi-Region 복제**: 리전 단위 재해 대비
- **Auto Scaling**: 트래픽 급증 대응
- **Health Check + Failover**: 비정상 리소스 자동 제외
- **백업 및 DR 계획**: 데이터 손실 방지

## 주요 서비스 SLA 비교

| 서비스 | 기본 SLA | Multi-AZ/리전 SLA | 주요 조건 |
|--------|----------|-------------------|-----------|
| EC2 | 99.5% (단일 인스턴스) | 99.99% (Multi-AZ) | Multi-AZ 배포 필수 |
| RDS | SLA 없음 | 99.95% (Multi-AZ) | Multi-AZ 배포 필수 |
| S3 | 99.9% | - | 단일 리전 내 자동 복제 |
| EBS | 99.99% | - | 단일 AZ 내 볼륨 |
| Lambda | 99.95% | - | 리전 단위 |
| DynamoDB | 99.99% | 99.999% (Global Tables) | Global Tables 활성화 시 |

## 정리

AWS SLA는 각 서비스의 월간 가동률을 보장하는 공식 계약으로, 기준 미달 시 월 사용 요금의 일정 비율을 서비스 크레딧으로 보상하며, EC2는 Multi-AZ 배포 시 99.99%, S3는 99.9%, RDS는 Multi-AZ 시 99.95% 가동률을 보장하고, 크레딧은 고객이 장애 발생 후 2개 청구 사이클 이내에 AWS Support를 통해 직접 청구해야 하며, SLA는 가용성만 보장하므로 성능, 데이터 손실, 사용자 오류는 고객 책임이며, 실제 고가용성 달성을 위해서는 Multi-AZ, Multi-Region, Auto Scaling, Health Check 등의 아키텍처 패턴을 적용해야 한다.
