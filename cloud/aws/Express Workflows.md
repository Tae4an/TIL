# AWS Express Workflows란

## 1. 개념 요약

AWS Express Workflows는 **AWS Step Functions에서 제공하는 워크플로 유형 중 하나로, 짧은 시간 안에 대량(고빈도)의 이벤트를 매우 빠르고 저렴하게 처리하기 위한 고속 워크플로 타입**이다.  
같은 Step Functions이지만, **오래 실행되는 복잡한 비즈니스 프로세스용인 Standard 타입과 달리, 수초~수분 이내에 끝나는 대량·실시간 처리에 최적화된 버전**이라고 보면 된다.

***

## 2. 특징

- **고성능·고처리량**  
  - 초당 수천~수만 건 수준의 실행을 처리하도록 설계되어, 이벤트 폭주 상황에서도 워크플로 오케스트레이션이 병목이 되지 않는다.  

- **짧은 실행 시간**  
  - 각 실행은 최대 수분(대략 5분 이내) 단기 실행에 적합하다.  
  - 장기간 대기·휴먼 승인·야간 배치 같은 워크로드는 Standard 워크플로를 사용해야 한다.

- **비용 구조 차이**  
  - Standard: “상태 전이(State transition)” 개수 기준으로 과금.  
  - Express: **실행 횟수 + 실행 시간(메모리 포함)** 기준으로 과금해, 대량·단기 워크로드에서 비용 효율이 높다.  

- **실행 시맨틱**  
  - 비동기 실행: 최소 1회(at-least-once) 실행 시맨틱.  
  - 동기 실행: 최대 1회(at-most-once) 실행 시맨틱.  
  - 즉, 간헐적인 중복 실행을 허용하는, 고처리량 이벤트 처리용에 맞게 설계되어 있다.

- **실행 이력 저장 방식**  
  - Standard: Step Functions가 장기간 상세 실행 이력을 내부에 저장·조회 가능.  
  - Express: 실행 이력을 CloudWatch Logs로 내보내고, 서비스 내부에는 짧게만 유지한다.

***

## 3. 어디에 쓰이는가

- **IoT 이벤트 처리**  
  - 수많은 센서/디바이스에서 들어오는 이벤트를 Lambda, Kinesis, DynamoDB, SNS 등과 조합해 빠르게 처리.  

- **스트리밍 데이터 처리·ETL**  
  - Kinesis, Kafka, EventBridge로 들어오는 이벤트를 실시간 필터링·변환·알림·집계하는 파이프라인에 사용.  

- **모바일/백엔드 API 오케스트레이션**  
  - API Gateway와 연동해, 한 번의 API 호출 안에서 여러 Lambda·외부 API·DB 호출을 짧은 시간에 순차/병렬 실행.  

- **마이크로서비스 통합**  
  - 동기 Express 실행으로 여러 마이크로서비스 호출, 에러 처리·재시도·분기 로직을 코드 대신 워크플로 정의로 관리.

***

## 4. Standard vs Express Workflows

| 항목 | Standard Workflows | Express Workflows |
|---|---|---|
| 주 사용처 | 장기 실행, 복잡 비즈니스 프로세스 | 대량, 단기, 고빈도 이벤트 처리 |
| 실행 최대 시간 | 최대 1년 | 최대 수분(약 5분) |
| 처리량 | 초당 수십~수백 수준 | 초당 수천~수만 실행 이상 |
| 과금 기준 | 상태 전이(State transition) | 실행 횟수 + 실행 시간·메모리 |
| 실행 시맨틱 | 거의 정확히 1회(Exactly-once에 가까움) | Async: At-least-once, Sync: At-most-once |
| 실행 이력 | Step Functions 내부 저장·조회 | CloudWatch Logs로 출력 |
| 적합 예시 | 주문 처리, 승인/결재, 배치 작업 | IoT, 스트림 처리, API 백엔드, 이벤트 팬아웃 |

***

## 5. 한 줄 정리

AWS Express Workflows는 **AWS Step Functions의 고속 워크플로 타입으로, 수분 이내에 끝나는 고빈도·대량 이벤트를 저렴한 비용으로 오케스트레이션하기 위한 “고처리량·단기 실행용 서버리스 워크플로”** 다.
