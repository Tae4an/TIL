# Amazon RDS Proxy란?

## 개념 정의

Amazon RDS Proxy는 애플리케이션과 RDS/Aurora 데이터베이스 사이에 위치하는 완전관리형 데이터베이스 프록시 서비스다. **Connection Pooling(연결 풀)** 을 통해 데이터베이스 연결을 재사용하고, 애플리케이션이 열고 닫는 수많은 연결을 프록시가 소수의 지속적인 연결로 통합하여 DB 부하를 줄이고 확장성을 향상시킨다.

쉽게 말해, "DB 앞에 서서 수많은 애플리케이션 연결을 받아 소수의 DB 연결로 모아주는 중개자로, Lambda 같은 서버리스 환경에서 DB 연결 폭증을 막아주는 서비스"다.

## 핵심 기능

- **Connection Pooling(연결 풀링)**  
  애플리케이션의 많은 연결을 소수의 DB 연결로 통합·재사용하여, DB 메모리·CPU 부하를 줄이고 "Too Many Connections" 오류를 방지한다.

- **자동 장애 조치(Failover)**  
  DB 장애 시 애플리케이션 연결을 유지한 채 예비 인스턴스로 자동 전환하여, 장애 복구 시간을 최대 66% 단축한다.

- **IAM 인증·Secrets Manager 통합**  
  DB 비밀번호를 코드에 하드코딩하지 않고, IAM 또는 Secrets Manager로 자격 증명을 관리하고 자동 회전할 수 있다.

- **Multi-AZ 고가용성**  
  여러 가용 영역에 자동 배포되어 단일 AZ 장애에도 서비스를 지속한다.

- **완전관리형**  
  프록시 인프라 프로비저닝·패치·확장을 AWS가 자동 처리하므로 운영 부담이 없다.

## 작동 방식

1. **애플리케이션 연결**: 애플리케이션이 RDS 엔드포인트 대신 RDS Proxy 엔드포인트로 연결
2. **연결 풀 확인**: 프록시가 사용 가능한 DB 연결 검색
3. **연결 재사용**: 기존 유휴 연결을 애플리케이션에 할당 (없으면 새로 생성)
4. **쿼리 실행**: 애플리케이션 쿼리를 DB로 전달
5. **연결 반환**: 애플리케이션이 연결 종료 시 프록시가 연결을 풀로 반환 (DB 연결은 유지)
6. **멀티플렉싱**: 수백 개 애플리케이션 연결을 소수의 DB 연결로 통합

### Connection Pooling 원리

**일반 방식 (프록시 없음)**  
- 1,000개 Lambda 동시 실행 = 1,000개 DB 연결 생성
- DB 메모리·CPU 과부하, "Too Many Connections" 오류 발생

**RDS Proxy 사용**  
- 1,000개 Lambda가 프록시로 연결
- 프록시는 50개 DB 연결만 유지
- Lambda 연결을 50개 DB 연결에 멀티플렉싱
- DB 리소스 절감, 성능 향상

## 주요 활용 사례

- **AWS Lambda 통합**  
  Lambda 함수가 동시에 수백~수천 개 실행될 때 각자 DB 연결을 생성하면 DB가 과부하되므로, RDS Proxy가 연결을 풀링하여 DB 부하를 줄인다. 함수 종료 후 연결 정리 코드가 불필요하며, 코드가 간결해진다.

- **서버리스 아키텍처**  
  API Gateway + Lambda + RDS 구조에서 트래픽 급증 시 Lambda 스케일링과 함께 DB 연결이 폭증하는 것을 방지하며, 연결 관리를 프록시에 위임한다.

- **Failover 시간 단축**  
  Multi-AZ RDS 장애 시 일반적으로 30~60초 소요되지만, RDS Proxy는 10~20초로 단축하며, 애플리케이션 연결은 유지되고 재시도가 자동 처리된다.

- **보안 강화**  
  애플리케이션 코드에 DB 자격 증명을 하드코딩하지 않고, IAM 인증 또는 Secrets Manager 자동 회전을 활용하며, TLS로 전송 중 암호화를 강제한다.

## 지원 데이터베이스

**RDS**  
- MySQL 5.6 이상
- PostgreSQL 9.6 이상
- MariaDB 10.2 이상
- SQL Server 2019 이상

**Aurora**  
- Aurora MySQL 5.6 이상
- Aurora PostgreSQL 9.6 이상

**지원하지 않음**  
- Oracle
- Aurora Serverless v1 (v2는 지원)

## 비용

- vCPU당 시간당 약 $0.015
- 일반적으로 db.r5.large 기준 월 약 $15~$20
- DB 연결 절감으로 DB 인스턴스 크기 축소 가능하여 전체 비용은 절감될 수 있음
