# AWS VIP(Virtual IP)란
## 1. 개념 요약
VIP(Virtual IP, 가상 IP)는 **특정 물리 서버나 네트워크 인터페이스(ENI)에 고정 귀속되지 않고, 논리적으로 존재하는 IP 주소**다.  
AWS 환경에서 VIP는 주로 **액티브-스탠바이 HA(고가용성) 구성에서 장애 발생 시 트래픽을 자동으로 다른 서버로 전환하거나, 로드밸런서 앞단의 대표 IP 역할**을 하는 데 사용된다.

***
## 2. AWS에서 VIP가 필요한 이유
일반적인 온프레미스 환경에서는 VRRP/HSRP 같은 프로토콜로 VIP를 서버 간에 자유롭게 이동시킬 수 있다.  
하지만 **AWS VPC는 서브넷 기반으로 관리되고, 각 서브넷은 단일 AZ에 귀속**되기 때문에 전통적인 방식으로 VIP를 이동시키는 것이 바로 되지 않는다.  
이를 해결하기 위해 AWS에서는 아래와 같은 방법으로 VIP를 구현한다.

***
## 3. AWS에서 VIP 구현 방식
- **라우트 테이블 업데이트 방식**
  - VIP를 VPC CIDR 외부의 별도 /32 IP 주소로 설정한다.
  - 장애 발생 시 Lambda·스크립트가 VPC 라우트 테이블의 해당 /32 경로를 스탠바이 서버의 ENI로 자동 업데이트한다.
  - 클라이언트는 동일한 VIP로 계속 접근하고, 뒤에서 라우팅만 바뀌는 방식이다.

- **Elastic IP(EIP) 전환 방식**
  - 액티브 서버에 붙어 있던 EIP를 장애 발생 시 스탠바이 서버에 재연결(reassociate)한다.
  - EIP 주소 자체가 VIP처럼 동작하지만, 전환 시간이 수십 초 정도 소요된다.

- **Overlay IP(Non-VPC CIDR IP) 방식**
  - VPC CIDR 범위 밖의 사설 IP를 VIP로 사용하고, 라우트 테이블로 ENI에 라우팅한다.
  - 다중 AZ 환경에서도 동작하며, Transit Gateway와 함께 멀티패스 로드밸런싱에도 활용된다.

- **AWS Load Balancer 활용 (권장)**
  - ALB/NLB가 자체적으로 VIP 역할을 수행한다.
  - DNS 이름 뒤에 자동으로 여러 IP를 가지며, 대상 그룹 내 서버들에 트래픽을 분산한다.
  - AWS 관리형 서비스라 별도 VIP 전환 스크립트 없이 고가용성이 자동 보장된다.

***
## 4. 주요 사용 사례
- **Active-Standby HA 구성 (예: Redis, DB, 방화벽)**
  - 액티브 인스턴스 장애 시 VIP를 스탠바이로 자동 전환해 서비스 연속성을 유지.

- **방화벽·네트워크 어플라이언스 HA**
  - Palo Alto, Fortinet 같은 가상 방화벽의 액티브-스탠바이 전환 시 VIP를 통해 트래픽 경로를 자동 변경.

- **On-Premises 연동 HA**
  - Direct Connect·VPN을 통해 온프레미스에서 들어오는 트래픽이 항상 동일한 VIP로 도달하도록 구성.

- **멀티패스 로드밸런싱**
  - Overlay IP 기반 VIP에 여러 ENI를 라우팅해 여러 VM/Pod로 트래픽을 분산.

***
## 5. VIP 구현 방식 비교
| 방식 | 전환 속도 | 다중 AZ 지원 | 복잡도 |
|---|---|---|---|
| 라우트 테이블 업데이트 | 수초 내 | 가능 | 중간 (Lambda/스크립트 필요) |
| EIP 재연결 | 수십 초 | 가능 | 낮음 |
| Overlay IP + 라우트 | 수초 내 | 가능 | 중간~높음 |
| ALB/NLB (권장) | 자동 (거의 즉시) | 기본 지원 | 낮음 (관리형) |

***
## 6. 한 줄 정리
AWS VIP는 **특정 ENI에 귀속되지 않는 논리적 가상 IP로, 라우트 테이블 업데이트·EIP 전환·Overlay IP 등의 방식으로 HA 페일오버 시 트래픽을 자동 전환하는 데 사용되며, 대부분의 경우 ALB/NLB가 관리형 VIP 역할을 대체한다.**
