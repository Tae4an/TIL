# AWS Route 53 라우팅 정책

## 개념 정의

Route 53 라우팅 정책(Routing Policy)은 하나의 도메인 이름에 여러 리소스(IP 주소, ALB, EC2 등)가 연결되어 있을 때, DNS 쿼리에 **어떤 리소스를 응답할지 결정하는 규칙**이다. 사용자 위치, 리소스 상태, 지연 시간, 가중치 등 다양한 기준으로 트래픽을 분산·라우팅하여, 고가용성, 성능 최적화, 장애 조치, 지역별 서비스 제공을 구현할 수 있다.

쉽게 말해, "www.example.com 을 여러 서버 IP 중 어디로 안내할지 결정하는 교통 신호 체계로, 사용자 위치·서버 상태·부하 분산 등에 따라 최적의 서버를 선택하는 정책"이다.

## 7가지 라우팅 정책

### 1. Simple Routing (단순 라우팅)

**개념**  
- 가장 기본적인 라우팅 방식
- 하나의 레코드에 하나 또는 여러 개의 IP 주소 지정
- 여러 IP가 있으면 Route 53이 무작위로 하나를 응답 (클라이언트가 선택)

**특징**  
- Health Check 지원 안 함
- 조건 없는 단순 DNS 응답

**사용 사례**  
- 단일 리소스를 가리키는 기본 DNS 설정
- 테스트·개발 환경

### 2. Weighted Routing (가중치 기반 라우팅)

**개념**  
- 지정한 가중치(비율)에 따라 트래픽을 여러 리소스로 분산
- 예: A 리소스 70%, B 리소스 30%

**특징**  
- Health Check 지원
- 가중치 0~255 사이 값 설정
- 로드 밸런싱, Blue/Green 배포, A/B 테스트에 활용

**사용 사례**  
- 신규 버전 점진적 배포 (10% 트래픽만 신규 버전으로)
- 리전 간 로드 밸런싱
- 서버 용량에 따른 트래픽 분산

### 3. Latency-Based Routing (지연 시간 기반 라우팅)

**개념**  
- 사용자와 가장 낮은 지연 시간(Latency)을 가진 AWS 리전의 리소스로 라우팅
- Route 53이 사용자 위치에서 각 리전까지의 네트워크 지연 시간을 측정

**특징**  
- Health Check 지원
- AWS 리전 기반 (리전별로 레코드 생성 필요)
- 성능 최적화에 중점

**사용 사례**  
- 글로벌 사용자에게 최적의 성능 제공
- 멀티 리전 애플리케이션의 응답 속도 개선

### 4. Failover Routing (장애 조치 라우팅)

**개념**  
- Active-Passive 구성
- Primary(주) 리소스가 정상이면 Primary로 라우팅
- Primary가 비정상이면 Secondary(보조) 리소스로 자동 전환

**특징**  
- Health Check 필수
- Primary/Secondary 역할 지정
- 재해 복구(DR)에 최적화

**사용 사례**  
- 주 서버 장애 시 백업 서버로 자동 전환
- 멀티 리전 DR 아키텍처
- 고가용성 확보

### 5. Geolocation Routing (지리적 위치 라우팅)

**개념**  
- 사용자의 지리적 위치(대륙, 국가, 주/도)를 기준으로 트래픽 라우팅
- "한국 사용자는 서울 리전으로, 미국 사용자는 버지니아 리전으로"

**특징**  
- Health Check 지원
- Default 레코드 설정 가능 (매칭되지 않는 위치)
- 위치 기반 콘텐츠 제공, 규정 준수에 활용

**사용 사례**  
- 국가별 콘텐츠 제공 (언어, 통화, 저작권 제한)
- 특정 국가 트래픽 차단 (규제 준수)
- 지역별 서비스 분리

### 6. Geoproximity Routing (지리적 근접 라우팅)

**개념**  
- 사용자와 리소스의 **지리적 근접성**을 기반으로 라우팅
- Bias(편향값)를 설정해 특정 리소스로 더 많은/적은 트래픽 유도 가능

**특징**  
- Route 53 Traffic Flow에서만 사용 가능
- Bias 값: +1~+99 (트래픽 범위 확대), -1~-99 (트래픽 범위 축소)
- AWS 리전 또는 위도/경도 좌표로 리소스 위치 지정

**사용 사례**  
- 특정 리전으로 트래픽을 더 많이 유도 (신규 데이터센터 테스트)
- 리소스 간 부하 조정

**Geolocation vs Geoproximity**  
- Geolocation: 사용자의 위치 기반 (국가·대륙 단위)
- Geoproximity: 사용자와 리소스 간 물리적 거리 기반 (Bias 조정 가능)

### 7. Multivalue Answer Routing (다중 값 응답 라우팅)

**개념**  
- Route 53이 최대 8개의 정상(Healthy) 리소스를 무작위로 응답
- 클라이언트가 여러 IP 중 하나를 선택하여 연결

**특징**  
- Health Check 필수
- ELB와 유사하지만 클라이언트 측에서 선택
- 간단한 로드 밸런싱 효과

**사용 사례**  
- 여러 웹 서버 간 간단한 로드 밸런싱
- DNS 레벨의 가용성 향상

### 8. IP-Based Routing (IP 기반 라우팅)

**개념**  
- 클라이언트의 **소스 IP 주소**를 기반으로 라우팅
- CIDR 블록별로 트래픽을 다른 리소스로 라우팅

**특징**  
- Health Check 지원
- IP 주소 범위(CIDR)를 레코드에 매핑

**사용 사례**  
- ISP별로 다른 엔드포인트 제공
- 특정 IP 대역의 트래픽 격리

## 라우팅 정책 비교표

| 정책 | 기준 | Health Check | 주요 사용 사례 |
|------|------|--------------|----------------|
| Simple | 없음 (무작위) | 불가 | 기본 DNS, 단일 리소스 |
| Weighted | 가중치 비율 | 가능 | A/B 테스트, Blue/Green 배포 |
| Latency | 지연 시간 | 가능 | 글로벌 성능 최적화 |
| Failover | Primary/Secondary | 필수 | 재해 복구, 고가용성 |
| Geolocation | 사용자 위치 | 가능 | 지역별 콘텐츠, 규정 준수 |
| Geoproximity | 지리적 근접성 + Bias | 가능 | 리소스 간 부하 조정 |
| Multivalue | 무작위 (최대 8개) | 필수 | 간단한 로드 밸런싱 |
| IP-Based | 소스 IP 주소 | 가능 | ISP별 라우팅, IP 격리 |

## 정리

Route 53 라우팅 정책은 DNS 쿼리에 대해 어떤 리소스를 응답할지 결정하는 규칙으로, Simple(기본), Weighted(가중치 분산), Latency(지연 시간 최소화), Failover(장애 조치), Geolocation(사용자 위치), Geoproximity(리소스 근접성+Bias), Multivalue(최대 8개 정상 리소스), IP-Based(소스 IP) 8가지 정책을 제공하며, Health Check와 연동하여 비정상 리소스를 자동 제외하고, 글로벌 트래픽 관리, 고가용성, 성능 최적화, 재해 복구, 지역별 서비스 제공, A/B 테스트 등 다양한 시나리오에 활용된다.
