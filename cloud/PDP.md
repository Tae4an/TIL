# PDP란?

## 1. PDP (Policy Decision Point)

PDP는 **접근 요청이 들어왔을 때, 정의된 정책과 각종 속성 정보를 바탕으로 “허용 / 거부 / 조건부 허용”을 결정하는 정책 판단 엔진**이다.  
XACML·제로 트러스트 아키텍처에서 “정책을 평가해 최종 결론을 내리는 두뇌” 역할을 한다.

- 하는 일  
  - PEP가 올린 “이 요청을 허용해도 될까?” 질의를 받아서  
  - 어떤 정책들이 적용되는지 찾고  
  - 필요한 속성(사용자, 디바이스, 리소스, 환경 등)을 PIP에서 조회한 뒤  
  - Allow / Deny / Obligations(조건·추가 요구사항) 형태로 결론을 내려 돌려준다.  

***

## 2. PEP (Policy Enforcement Point)

PEP는 **실제로 요청을 가로채서, PDP에게 물어보고, PDP의 결정을 “집행(enforce)”하는 지점**이다.

- 위치 예시  
  - API Gateway, 리버스 프록시, ZTNA 게이트웨이, 서비스 메시 사이드카, 애플리케이션 내 필터/미들웨어 등.  

- 하는 일 플로우  
  1. 클라이언트의 접근 요청을 인터셉트  
  2. 요청 정보를 XACML 등 포맷으로 변환해 PDP에 질의  
  3. PDP 응답(허용/거부/조건부 허용)을 수신  
  4. 응답에 따라  
     - 세션 생성 / 요청 전달  
     - 차단(403 등)  
     - 추가 인증 요구, 리다이렉트 등 조치 수행  

***

## 3. PIP (Policy Information Point)

PIP는 **PDP가 정책을 평가할 때 필요한 “속성·컨텍스트 데이터”를 제공하는 정보 소스**이다.  
쉽게 말해, “판단을 내리려는 심판(PDP)이 참고하는 선수·환경 정보 DB” 역할을 한다.

- 제공하는 정보 예시  
  - 사용자 속성: 소속 조직, 직무, 권한, 계정 상태  
  - 디바이스 속성: OS, 보안 상태, 관리 여부  
  - 리소스 속성: 분류(기밀/일반), 소유 부서, 태그  
  - 환경 속성: 접속 위치, 시간대, 리스크 레벨, MFA 여부 등  

- 구현 예  
  - AD/LDAP, HR 시스템, CMDB, MDM, 자산관리, Threat Intel, 리스크 엔진 등과 연동해 PDP 질의에 필요한 속성을 반환.  

***

## 4. PAP (Policy Administration Point)

PAP는 **관리자가 접근 제어 정책을 “정의·관리·배포”하는 지점**, 즉 정책 관리 콘솔/서비스에 해당한다.

- 하는 일  
  - 정책 작성: 누가(Subject)가 무엇(Resource)에 언제/어디서/어떻게(Action, Condition) 접근 가능한지 규칙 정의  
  - 정책 검증·버전 관리: 테스트, 검토, 승인, 롤백 등  
  - 정책 저장·배포: 정책 저장소(PRP 등)에 보관하고, PDP로 배포·동기화  

- 비유  
  - PAP: 룰북을 만드는 곳(규칙 작성·관리)  
  - PDP: 그 룰북을 보고 판정하는 심판  
  - PEP: 심판 판정대로 문 앞에서 실제로 막거나 들이는 문지기  
  - PIP: 심판이 판정 전에 참고하는 선수/환경 정보 DB  

***

## 전체 흐름 요약

1. 관리자가 PAP에서 정책을 정의하고 저장소에 배포한다.  
2. 클라이언트가 리소스에 접근 → PEP가 요청을 가로챈다.  
3. PEP가 PDP에 “이 요청 허용해도 되냐?”고 질의한다.  
4. PDP는 필요한 속성을 PIP에서 조회해 정책을 평가하고, 허용/거부 결정을 반환한다.  
5. PEP는 그 결정에 따라 요청을 허용·차단·조건부 허용한다.  

이 네 가지(PDP, PEP, PIP, PAP)를 분리하면, 제로 트러스트·XACML 기반의 **확장 가능한 중앙 정책 관리 + 분산 집행** 구조를 만들 수 있다.
