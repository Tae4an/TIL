# DSR(Direct Server Return)이란?

## 개념 정의

DSR(Direct Server Return)은 **로드밸런서가 클라이언트의 요청(Request)만 받아서 서버로 전달하고, 서버의 응답(Response)은 로드밸런서를 거치지 않고 클라이언트에게 직접 보내도록 하는 로드밸런싱 동작 방식**이다.  
이름 그대로, 서버가 결과를 “직접(direct) 클라이언트에게 리턴(return)”하기 때문에, 응답 트래픽이 로드밸런서를 우회한다.  

***

## 왜 쓰는 방식인가

일반적인 L4/NAT 방식에서는 흐름이 이렇게 된다.  
클라이언트 → 로드밸런서 → 서버 → 로드밸런서 → 클라이언트  

DSR에서는 이렇게 바뀐다.  
클라이언트 → 로드밸런서 → 서버 → 클라이언트  

즉,  
- 요청(Request)만 로드밸런서를 지나고  
- 응답(Response)은 서버가 곧바로 클라이언트로 보내므로  

다음과 같은 효과가 있다.  
- 응답 트래픽(보통 용량이 크다)이 로드밸런서를 통과하지 않아, 로드밸런서 대역폭·CPU 부하 감소  
- 고용량 응답(대용량 파일 다운로드, 동영상 등) 환경에서 로드밸런서 병목 완화  

***

## 동작 핵심 포인트

- 로드밸런서는 **목적지 IP(VIP)는 그대로 두고**, L2/L3 수준에서 서버로 패킷을 전달한다.  
- 서버에는 로드밸런서의 VIP를 자신(루프백 등)에 바인딩해 두고, VIP로 온 요청을 자기 것으로 인식한다.  
- 서버가 응답 패킷을 만들 때, 출발지 IP를 VIP(서비스 IP)로 하여 **클라이언트에게 직접 전송**한다.  
- 그 결과, 클라이언트 입장에서는 요청을 보낸 IP(로드밸런서 VIP)에서 응답이 오는 것처럼 보이지만, 실제로는 서버가 직접 보내는 구조다.  

***

## 장단점

**장점**

- 로드밸런서가 응답 트래픽을 처리하지 않아 **처리량(Throughput)·성능 향상**  
- 고트래픽 서비스에서 로드밸런서 병목과 비용 감소  
- 클라이언트 IP를 서버에서 직접 볼 수 있는 구성도 용이 (설정 방식에 따라)  

**단점**

- 서버에 VIP 설정, 라우팅 등 네트워크 구성이 일반 NAT 방식보다 복잡  
- L4 로드밸런싱에는 잘 맞지만,  
  - SSL 종료/재암호화  
  - 응답 내용 기반 L7 로직  
  같은 기능은 로드밸런서가 응답을 보지 못하므로 구현이 어렵거나 불가능  

***

## 언제 쓰는지

- 응답 데이터(Back data)가 많고, 로드밸런서가 병목이 되기 쉬운 서비스  
  - 예: 대용량 파일 다운로드, 스트리밍, 이미지/미디어 전송 등  
- L7 콘텐츠 조작이 크게 필요 없고, **L4 레벨 Load Balancing + 최대 성능**이 우선인 환경  
- One-arm 구조(로드밸런서가 인라인이 아니라 한쪽 팔처럼 붙어 있는 구조)에서 자주 사용되는 모드  

정리하면, DSR은 “요청만 로드밸런서, 응답은 서버 → 클라이언트 직행” 구조로,  
고트래픽 환경에서 로드밸런서 부하를 줄이고 성능을 극대화하기 위한 L4 로드밸런싱 모드라고 보면 된다.
