# ZTNA(Zero Trust Network Access)란?

## 개념 정의

ZTNA(Zero Trust Network Access)는 **“절대 신뢰하지 않고 항상 검증한다”는 제로 트러스트 원칙을 원격·내부 접근에 적용하는 접근 제어 모델**이다.  
기존처럼 VPN으로 내부망 전체를 열어주는 것이 아니라, 사용자·디바이스·맥락을 매번 검증한 뒤 **특정 애플리케이션 단위로 최소 권한만** 접근을 허용하는 방식임.  

***

## ZTNA가 하는 일 한 줄로

“사용자와 기기가 어디에 있든, ‘누구인지·얼마나 안전한지·지금 상황이 어떤지’를 매번 확인하고, **딱 그 사람이 필요로 하는 앱만 골라서 열어주는 원격 접속/내부 접근 방식**”이라고 보면 된다.  

***

## 핵심 개념

- 기본 거부(Default Deny)  
  - 네트워크 안에 있든 밖에 있든 기본은 “접근 불가” 상태에서 시작.  
  - 요청마다 신원·디바이스·컨텍스트를 검증한 뒤 필요한 리소스에만 허용.  

- 최소 권한(Least Privilege)  
  - 사용자 업무에 필요한 애플리케이션/리소스만 접근 허용.  
  - 한 계정이 유출되더라도 피해 범위를 최소화하는 구조.  

- 애플리케이션 단위 접근(App-centric)  
  - 네트워크 세그먼트 전체가 아니라, 앱·서비스 단위로 접근 권한 부여.  
  - 네트워크 상의 측면 이동(Lateral Movement)을 어렵게 만듦.  

- 컨텍스트 기반 정책  
  - 사용자 ID, 디바이스 보안 상태(패치·EDR 유무 등), 위치, 시간대, 리스크 수준 등을 종합해 정책 평가.  
  - 위험도가 높으면 MFA 요구, 읽기 전용, 접근 차단 등 동적으로 조정.  

***

## ZTNA와 VPN 차이

| 항목 | VPN | ZTNA |
|------|-----|------|
| 접근 범위 | 내부 네트워크 전체/대부분에 접속 터널 제공 | 특정 애플리케이션·서비스만 선택적 허용 |
| 신뢰 모델 | 터널 열리면 내부는 상대적 신뢰 | 항상 검증, 기본 불신(Zero Trust) |
| 단위 | 네트워크 단위(IP/서브넷) | 애플리케이션·사용자·기기·컨텍스트 단위 |
| 보안 위험 | 계정/단말 탈취 시 내부망 측면 이동 쉬움 | 최소 권한·마이크로 세분화로 확산 억제 |
| 아키텍처 | 중앙 VPN 게이트웨이에 트래픽 집중 | 분산된 ZTNA 게이트웨이/서비스, 클라우드 네이티브 구조가 많음 |

***

## 동작 방식 (흐름)

1. 사용자가 ZTNA 포털/클라이언트에 접속 → 사용자 인증(MFA 포함).  
2. 클라이언트/에이전트가 디바이스 보안 상태(OS 버전, 패치, EDR 유무 등) 평가.  
3. ZTNA 컨트롤러가 ID·디바이스·위치·시간 등 컨텍스트를 기반으로 정책을 평가.  
4. 허용된 애플리케이션 목록만 사용자에게 표시 또는 연결 허용.  
5. 사용자가 특정 앱 요청 시, ZTNA 게이트웨이가 그 앱에 대해서만 TLS 기반 직접 연결을 설정.  
6. 세션 중에도 이상 행위가 감지되면 추가 인증 요구, 권한 축소, 세션 종료 등 조치.  

***

## 어디에 쓰는지 (주요 활용)

- 재택·원격 근무 환경에서 VPN 대신 **제로 트러스트 기반 안전한 원격 접속** 제공.  
- 클라우드·온프레미스에 흩어진 내부 업무 시스템(ERP, 인트라넷, 개발 시스템 등)에 대한 사용자·파트너 접근 통제.  
- 기존 “경계형 보안(내부=신뢰, 외부=비신뢰)” 모델에서,  
  “내부·외부 모두 검증, 최소 권한” 구조로 전환하려는 제로 트러스트 전략의 핵심 구성요소로 사용됨.
