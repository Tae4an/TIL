# Single-Pass Inspection이란?
## 개념 정의
Single-Pass Inspection(또는 Single-Pass Architecture)은 **트래픽을 한 번만 스캔하면서, 그 과정에서 필요한 모든 보안·네트워크 기능(앱 식별, 사용자 식별, 콘텐츠/위협 검사, 정책 적용 등)을 동시에 수행하는 처리 방식**을 의미한다.  
기존처럼 기능별 엔진을 여러 번 거치는 multi‑pass 구조가 아니라, “패킷/플로우를 한 번 읽으면서(App‑ID, User‑ID, Threat, URL 등) 전부 판단하고 처리하는” 아키텍처라고 보면 된다.  

***
## 왜 나왔는지 (배경)
전통 UTM/NGFW는 기능이 늘어날수록 처리 구조가 다음과 같이 되는 경우가 많다.

- 1차 패스: 방화벽(포트/IP) 검사  
- 2차 패스: 애플리케이션 식별  
- 3차 패스: IPS/안티바이러스  
- 4차 패스: URL 필터링 …  

이렇게 “여러 엔진을 순차적으로 반복 통과”하는 구조는 기능을 켤수록 지연·CPU 부하·처리량 저하가 커지는 문제가 있음.  
Single‑Pass Inspection은 이 문제를 풀기 위해, **단일 엔진/단일 패스에서 풀스택 분석을 끝내도록 설계된 구조**라고 이해하면 됨.  

***
## 동작 개념 (한 번에 하는 것들)
Single‑Pass 구조에서는 한 번의 패킷/스트림 처리 과정에서 대략 이런 일을 동시에 수행한다.

- 세션/스트림 분석: 흐름 단위 상태 추적  
- 애플리케이션 식별(App‑ID 등)  
- 사용자/디바이스 식별(User‑ID 등 디렉터리 연동)  
- 컨텐츠/위협 검사  
  - IPS/시그니처 매칭  
  - 안티바이러스/멀웨어 탐지  
  - URL/카테고리 필터링  
  - 파일·데이터 패턴 검사(DLP 연계 등)  
- 정책 결정 및 로그/메타데이터 생성  

핵심은 “DPI(Deep Packet Inspection)를 여러 번 반복하지 않고, 한 번 심층 분석한 결과를 공유해서 모든 보안·정책 결정을 끝낸다”는 점이다.  

***
## 장점
- 성능  
  - 패킷을 여러 번 읽고 다시 검사하지 않으므로 CPU·메모리 부담 감소, 지연(Latency) 축소.  
  - 고부하 상황에서도 보안 기능을 모두 켠 상태로 높은 처리량 유지에 유리.  

- 일관성  
  - 한 번의 분석 결과(앱, 사용자, 컨텐츠 정보)를 모든 기능이 공유하므로,  
    정책 일관성이 높고, 서로 다른 엔진 간 판단 불일치가 줄어든다.  

- 확장성  
  - 새로운 보안 기능을 추가해도, 같은 Single‑Pass 엔진 안에서 재사용할 수 있어 구조 확장에 유리함.  

***
## 어디서 많이 쓰이는 개념인가
- 차세대 방화벽(NGFW), UTM, SD‑WAN/SASE/SSE 플랫폼에서 “고성능 DPI + 풀스택 보안”을 구현할 때 핵심 설계 개념으로 사용된다.  
- Palo Alto Networks의 **SP³(Single‑Pass Parallel Processing)**, 여러 벤더의 “Single‑Pass DPI/Inspection/One‑Pass Architecture” 같은 명칭으로 많이 언급된다.  

정리하면, Single‑Pass Inspection은 **“한 번 스캔할 때 다 보고, 다 판단하고, 다 막는” 방식으로 성능과 보안을 동시에 노리는 NGFW/SASE 계열 아키텍처 개념**이라고 볼 수 있다.
